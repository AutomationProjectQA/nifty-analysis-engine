# Nifty Analysis Engine – Complete Implementation Plan (V2)

An Agentic AI system that continuously analyzes Nifty spot/futures, option chains, PCR, OI, VIX, and market sentiment to identify high-probability option buying opportunities, sending notifications only when confidence meets or exceeds a strict 80% threshold.

---

## Core Design Principles

1. **Data First:** No signal generation without historical data.
2. **Deterministic Confidence:** Confidence is calculated mathematically, not generated by AI.
3. **LLM Explains, Not Decides:** The trading engine remains profitable even if the LLM is deactivated.
4. **Multi-Layer Validation:** Every signal must pass through all specialized agent layers.

---

## Technical Stack

- **Backend:** Java 21, Spring Boot 3.5
- **Database:** PostgreSQL 16
- **Cache:** Redis 7
- **AI:** GPT-5, Gemini, Spring AI / LangChain4j
- **Notification:** Telegram Bot API
- **Infrastructure:** Docker, Docker Compose

## 9-Sprint Execution Roadmap

### Sprint 1 — Data Collection (Core Foundation)
- Setup Spring Boot, Flyway, JPA, Postgres, Redis, and WebClient.
- Write Flyway V1 migration schema.
- Define JPA entities matching all V2 tables: `MarketSnapshot`, `MarketCandle`, `OptionSnapshot`, `TradeSignal`, `SignalExplanation`, `TradeResult`, `ConfidenceWeight`.
- Create `MarketScheduler` triggering every 1-5 minutes during market hours.
- Implement `MarketDataClient` & `OptionChainClient` interfaces.
- Create `SimulatedDataClient` providing high-quality, realistic market tick streams.
- Persist snapshots in Postgres, and store current values in Redis (`MARKET:LATEST`, `OPTION:LATEST`).
- Create controller endpoints to verify data collection status.

### Sprint 2 — Technical & Options Analytics
- Implement technical indicator processors:
  - Calculate EMA20 & EMA50.
  - Calculate RSI (14).
  - Calculate VWAP using intraday prices and cumulative volume.
- Implement option chain processors:
  - Put-Call Ratio (PCR) trend calculation.
  - Max Pain calculation.
  - Open Interest (OI) Build-up Detection per strike:
    - **Long Build-up**: Price rising, OI rising.
    - **Short Covering**: Price rising, OI falling.
    - **Short Build-up**: Price falling, OI rising.
    - **Long Unwinding**: Price falling, OI falling.

### Sprint 3 — Market Context Layer (Context Agents)
- **Market Regime Agent:** Classifies market into *Trending Bullish*, *Trending Bearish*, *Sideways*, *High Volatility*, or *Event Driven*. Helps avoid trading options in sideways ranges.
- **Multi-Timeframe Agent:** Validates structural alignment across 5m, 15m, 30m, and 60m timeframes.
- **Sentiment Agent:** Fetches global metrics (GIFT Nifty, Dow/Nasdaq Futures, DXY, Crude) and daily FII/DII data.
- **Liquidity Agent:** Filters out illiquid strikes using bid-ask spreads, volume, and minimum OI.
- **Event Risk Agent:** Tracks economic calendar (RBI, Fed, CPI) and flags impending volatility.

### Sprint 4 — Confidence Engine (Deterministic Scoring)
- Initialize the `confidence_weight` table with baseline values (Trend=20, OI=20, PCR=15, VWAP=15, RSI=10, Futures=10, Sentiment=10).
- Create `ConfidenceEngine` service:
  - Fetches weight factors dynamically from the database.
  - Evaluates inputs from Sprint 2 & 3 agents.
  - Produces a final mathematical score from 0% to 100%.
  - Restricts signal creation unless `Confidence >= 80%`.

### Sprint 5 — Critic Engine (Trade Invalidation)
- Create `CriticAgent` service:
  - Challenges candidate trade signals with potential invalidation criteria.
  - Applies negative score adjustments (e.g., -15% confidence for high Call writing concentration, -10% for RSI divergence, -10% for weak breakout volume).
  - Saves all adjustments into the `signal_explanation` table.

### Sprint 6 — Telegram Notification Service
- Implement Telegram bot notification integration via `TelegramBotService`.
- Formats and sends option buy signals containing Strike, Entry price, SL, Targets, Confidence score, and a list of validated technical/fundamental reasons.

### Sprint 7 — Backtesting Engine
- Implement `BacktestingEngine` service:
  - Replays historical candles and snapshots from the database.
  - Simulates the entire agent analysis and entry-timing validation lifecycle.
  - Tracks signal performance: accuracy, profit/loss, average holding time, and drawdowns.
  - Writes test results to the `trade_result` table.

### Sprint 8 — LLM Decision & Explanation Layer
- Add LangChain4j or Spring AI configurations to integrate with Gemini or GPT.
- **Decision Agent:** Collects structural data and agent scores. Sends this to the LLM to write a natural language explanation of the trade's thesis and format the Telegram message. 
- *Crucially, the LLM does not decide entry/SL/targets or confidence scores—it only translates the engine's deterministic data into natural language.*

### Sprint 9 — Self-Learning Model (Adaptive Optimizations)
- Implement `AdaptiveWeightsService`:
  - Periodically parses `trade_result` history.
  - Correlates individual agent success scores with successful trades.
  - Adjusts parameters in the `confidence_weight` table to favor highly-predictive indicators and demote underperforming ones.
